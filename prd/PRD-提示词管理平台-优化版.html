Chapter 01

需求背景

当前系统面临的问题与挑战

旧系统维护困难

原提示词平台基于Vibe Coding快速搭建，代码质量差、BUG频发。随着功能迭代，技术债务累积到难以维护的程度，严重影响团队效率。

复制粘贴易出错

AI Infra应用管理的提示词配置是"文本框填写"模式。研发需要从旧平台调试好、复制、粘贴到应用配置，流程繁琐且容易出错。

变量调试效率低

提示词调试时需要反复修改变量值进行测试。每次都要手动输入变量，无法复用评测集中的测试数据，批量测试效率极低。

Chapter 02

解决方案

系统性的产品方案设计

核心方案：重建提示词管理平台

采用全新架构重建提示词管理系统，实现提示词的全生命周期管理，并与AI Infra应用管理深度打通。

方案要点

提示词即服务

提示词作为一等公民，支持完整的生命周期管理（创建、编辑、调试、发布、归档）

与应用管理打通

AI Infra可直接选用已调试好的提示词，告别复制粘贴，降低出错率

评测集变量引用

调试时可直接引用评测集变量，支持批量测试，大幅提升调试效率

交互原型预览

页面结构

提示词列表页采用经典的"筛选区 + 列表区 + 操作区 + 分页区"四区布局，各区域职责明确。

区域 功能 规则说明
筛选区 多维度检索提示词 • 级联筛选：项目与应用采用级联选择器，选择项目后自动加载该项目下的应用列表
• 关键词搜索：支持按提示词名称、负责人进行模糊搜索，输入时实时触发搜索
列表区 展示提示词核心信息（ID、项目、应用、名称、描述、更新时间、负责人） • 排序规则：默认按最近更新时间倒序排列（时间越晚在越前面）
• ID 生成：自增主键，删除后不回滚
• 权限控制：列表内容基于当前用户权限过滤，只展示可见范围内的提示词
操作区 每行数据末尾提供快捷操作入口 • 编辑：点击后跳转到提示词编辑页
• 复制：以"原名称-副本"为名称创建新提示词，复制原有的 Prompt 内容和配置
• 删除：点击后弹出二次确认对话框，确认后删除该提示词
分页区 列表底部提供分页导航 • 默认每页展示 10 条数据
• 页码区显示总页数，支持上一页/下一页/首页/尾页快捷跳转
• 支持直接输入页码跳转到指定页

6.2.1 基础信息配置

页面顶部配置提示词的基础信息，包含以下字段：

字段 控件类型 规则说明
项目 * 级联选择器 从系统项目列表中选择，选择后自动加载该项目下的应用列表
应用 * 级联选择器 依赖项目选择；项目未选择时为禁用状态；支持搜索过滤应用名称
名称 * 文本输入 长度 2-50 字符，全局唯一；输入框失焦时校验唯一性，名称重复时提示"该名称已存在"
描述 文本域 长度 0-200 字符；不显示剩余字数提示；输入长度达到上限后无法继续输入
负责人 * 文本输入 创建人默认写入为负责人；支持多个负责人，用英文逗号 , 分隔；前端自动将中文逗号 ， 转译为英文逗号，并去除多余空格后再保存

6.2.2 Prompt编辑器（左栏）

左栏为 Prompt 内容编辑区域，采用卡片式设计，支持多角色消息配置。

功能 规则说明
消息卡片结构 卡片顺序固定为：System → User → Assistant（可多个）
• System Prompt：默认卡片，不可删除，始终展示在最前面
• User Prompt：默认卡片，不可删除，位于 System 卡片之后
• Assistant Prompt：可动态添加，支持添加多个，每个卡片右上角提供删除按钮
添加消息 点击"添加 Assistant Prompt"按钮，系统在最后一个卡片之后插入新的 Assistant 卡片。可连续添加多个。
编辑内容 在卡片内的文本域中编辑消息内容：
• 文本域根据内容高度自动增高
• 变量占位符（如 {{variable_name}}）高亮显示，便于识别
• 支持 Tab 键缩进，方便编辑格式化内容
添加变量占位符 点击卡片内的 + 变量 按钮（位于右下角），弹出变量类型选择浮层：
• 支持插入三种类型：文本(text)、图片(image)、文件(file)
• 插入后在光标位置生成占位符，格式为 {{variable_name}}
• 系统实时扫描左栏内容，自动提取所有符合格式的占位符，去重后同步到右栏"变量输入"区域
变量名双向联动 左栏与右栏的变量名实现双向同步：
• 在左栏修改占位符名称，右栏对应变量名自动更新
• 在右栏修改变量名，系统自动回写左栏所有消息中的同名占位符
• 变量命名规则：必须以字母开头，只能包含字母、数字、下划线
删除卡片 仅 Assistant 卡片右上角显示删除按钮，点击后移除该消息卡片。System 和 User 卡片不显示删除入口。删除卡片时，该卡片内的变量占位符从左栏移除，右栏对应的变量输入项同步移除。

6.2.3 节点配置（中栏）

中栏配置模型运行参数，控制 AI 的推理行为和输出特性。

参数 控件类型 规则说明
模型选择 下拉选择器 自动选中当前项目配置的默认模型；展示模型名称、版本号、价格信息（输入/输出 token 单价）
深度思考 单选按钮 配置模型的思考模式，支持三种选项：
• 开启：强制启用深度思考模式
• 关闭：强制关闭深度思考模式
• 自动：由模型根据输入自动决定是否启用
切换后即时生效，无需保存即可在下次运行时应用
温度 数值输入框 默认值 0.3；取值范围 0-2；必填；超出范围时阻止保存和运行，并提示错误
Seed 数值输入框 控制输出的可重现性，相同 Seed 在相同输入下产生相同输出；必填；支持数值校验
MaxTokens 数值输入框 限制模型最大输出 token 数量；必填；支持数值校验
Timeout 数值输入框 设置请求超时时间；单位：秒
Extra Body JSON 文本域 透传额外的请求参数到模型接口；需符合 JSON 格式，运行时会透传给模型

6.2.4 预览调试（右栏）

右栏是调试运行区域，包含运行结果展示。变量引用模式配置和变量输入均位于左栏。

引用评测集模式

在编辑提示词页面，可以选择引用已有的评测集来填充变量值：

  • 选择评测集：不使用简单下拉框，改为“评测集选择弹窗”。弹窗支持按评测集名称/ID/项目应用维度检索，确认后回填到右栏。
  • 渲染范围：评测集候选范围受“当前项目/应用 + 变量字段兼容”双重约束，仅展示可直接填充当前变量的评测集。
  • 选择用例：选择具体用例后，系统自动将该用例的输入变量填充到右栏变量输入区
  • 手动修改：自动填充后，仍可手动修改变量值进行调试，修改不影响原评测集数据
  • 持久化：保存提示词时，记录 ref_mode=eval、eval_set_id、case_id，下次进入时自动回填
自定义变量模式

不引用评测集时，使用自定义变量进行调试：

  • 自动提取变量：左栏编辑 Prompt 时，系统自动识别 {{variable_name}} 格式的占位符，去重后实时同步到右栏。占位符删除时，对应输入项自动移除。
  • 变量类型：支持三种类型 
    —— 文本(text)显示多行输入框；
    —— 图片(image)和文件(file)显示只读输入框+上传按钮
  • 双向联动：左栏修改占位符名称，右栏同步更新；右栏修改变量名，回写左栏所有消息中的同名占位符
  • 类型切换：支持 text/image/file 互切，切换后立即更换控件类型。⚠️ 从文件类切换时，会清空旧上传值以避免类型污染
  • 删除变量：在右栏点击变量项删除按钮，删除后同步清理左栏占位符、右栏输入值和上传文件引用
  • 持久化：保存提示词时，记录 ref_mode=custom 与变量快照（变量名、类型、当前值）。图片/文件变量保存元信息引用（文件名、URL/资源ID）
调试运行

使用当前变量值进行调试运行：

  • 单条运行：点击运行按钮，使用当前变量值调用模型。运行中显示加载态，完成后回填模型响应、模型输出、Trace性能指标。再次运行覆盖上次结果。
  • 复制结果：提供两个复制按钮 —— "复制(模型响应)"仅复制模型响应框内容；"复制(模型输出)"仅复制模型输出框内容
  • 批量运行：点击"批量运行"按钮跳转到评测中心，URL 参数带入 prompt_id 和 eval_set_id，评测中心自动填充筛选条件
注意

仅运行未保存，不视为最终状态。只有点击"保存"后，调试态才会被持久化。重新进入编辑页时默认回填上次状态，若变量结构变化，同名变量优先回填，缺失变量忽略，新变量按空值初始化。

6.2.5 同步到评测集

交互原型预览

功能说明

将当前运行结果通过浮层同步到评测集用例。

交互规则

  • 入口限制：仅编辑页提供该能力；新建页不提供"同步到评测集"入口
  • 显示条件：入口按钮名称为"同步到评测集"；仅在运行完成且存在输出后显示
  • 模式约束（强规则）：ref_mode=custom 仅允许“新增用例”；ref_mode=eval 仅允许“覆盖用例”。浮层中禁用不允许的模式，不可手动切换。
  • 评测集范围：custom+add 仅展示当前项目/应用下字段兼容的评测集；eval+overwrite 默认锁定当前已选评测集（如无已选则选第一个可用项），可更换用例。
  • 浮层字段：评测集、用例id、变量预览、期望输出（reference_output）

新增用例

仅在 ref_mode=custom 下可用。在目标评测集新增一行，用当前变量值写入输入字段，模型输出写入 reference_output。

覆盖用例

仅在 ref_mode=eval 下可用。仅覆盖选中用例的 reference_output，不覆盖变量输入字段。

6.2.6 评测集创建与变量演进

评测集创建流程

  • 步骤1：先选择项目/应用，作为评测集作用域。
  • 步骤2：选择评测集类型（提示词 / 应用）。
  • 步骤3：选择评测对象。提示词类型与应用类型均通过“对象选择弹窗”完成，不使用简单下拉框。

变量初始化与同步逻辑

  • 初始化：基于提示词或应用创建评测集时，按目标对象变量初始化评测集变量元数据（name/type/status/source/updatedAt）。
  • 新增变量：目标对象新增变量时，评测集同步新增变量，状态为 active，并在全部用例补空值。
  • 删除变量：目标对象删除变量时，评测集执行软删除（status=soft_deleted），默认不展示但保留历史数据。
  • 修改变量：按“新增变量”处理（旧变量软删除 + 新变量新增），不做自动 rename 合并。
  • 适用范围：上述规则对提示词型评测集与应用型评测集统一生效。

设计理念

提示词采用"当前态"设计，无显式版本号。应用管理仅把提示词作为模板来源，保存后转为应用内独立快照，避免提示词变更直接冲击线上应用。

核心规则

提示词层面：提示词本身没有版本号概念，只有"当前内容"。每次编辑保存后，提示词中心的当前内容立即更新；应用侧是否采纳该变化，取决于是否执行“重新从模板拉取”。

应用层面：应用配置保存时记录 template_prompt_id、template_prompt_version 与 resource_prompt_snapshot。后续应用内修改仅更新快照，提示词管理里的后续变更不会自动覆盖已保存应用。

版本跟踪：应用版本不仅记录资源增删，还记录资源内部变更摘要（提示词内容、模型参数、超时等），用于线上问题回溯。

典型场景

编辑提示词：直接修改当前内容，仅影响提示词中心“当前态”。已保存到应用内的资源快照不自动变化。

重新拉取模板：应用侧如需同步模板变更，需显式执行“重新从模板拉取”，并在保存后形成新的资源内部修订记录。

问题排查：当线上出现问题时，通过变更日志查看"谁在什么时候改了什么"。变更日志用于审计追溯，记录操作人、操作时间、变更内容摘要，但不存储完整的内容快照。

设计取舍

放弃的能力：精确回滚到某个历史版本。如果需要回滚，只能手动编辑恢复内容。
换来的是：无需存储历史快照、无版本对齐问题、系统简单直接。
适用场景：提示词/知识库更新频繁、回滚需求少、发布前有严格验证流程的团队。
