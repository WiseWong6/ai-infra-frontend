Trace 列表指标与详情摘要优化

本 PRD 仅覆盖本轮 Trace 追踪新增的展示优化：列表指标列、详情摘要位置与样式、过程日志节点头部指标，以及长日志展开/收起交互。线上已有的过程日志基础逻辑不作为本次新增需求重复定义。

Chapter 01

需求背景与目标

本次只处理“新增展示优化”，不重做 Trace 全链路交互定义

当前痛点

当前 Trace 列表对关键指标的呈现不够集中，TraceId、状态、耗时、Token、成本需要跨列扫读；详情弹窗顶部摘要区视觉过重，主视线容易被抢走；过程日志节点头部缺少一眼可扫的成本/耗时指标，长日志展开后收起入口不稳定，影响连续排障效率。

列表更适合扫读

把状态、TraceId 与关键指标组织到更符合排障习惯的列顺序中，降低横向比对成本。

详情头部降权

让摘要区服务信息获取，而不是压过详情正文与过程日志本身。

节点级指标更直观

在节点标题区直接给出状态、耗时、Token 与成本，减少进出内容区的切换。

长日志收放更稳定

让用户在任何滚动位置都能找到收起入口，避免超长日志带来的交互中断。

Chapter 02

范围界定

明确哪些是本轮新增，哪些只是沿用线上现状

本次纳入需求
  • Trace 列表字段顺序与展示方式调整。
  • Trace 详情顶部摘要区的位置与视觉层级调整。
  • 过程日志节点头部指标展示增强。
  • 长日志展开/收起交互优化。
沿用既有逻辑，不在本次新增中展开
  • 过程日志无目录/无树状链的线上主方案。
  • 扩展 ID 区与整体滚动结构。
  • 节点元信息行的真实日志字段规则。
  • 默认超过 3 行折叠的既有逻辑。
  • 自定义列能力，本轮暂不纳入。

Chapter 03

Trace 列表指标增强

让列表更适合横向比对与快速定位异常记录

核心变化

状态 icon 与 TraceId 合并成同一列，时间列前置，端到端耗时、总 Token、总成本进入固定列结构，并按排障顺序排在输入/输出之后。

模块 规则
时间列 Trace 创建时间前置，成为列表最先扫读的关键信息列。
TraceId 列 成功/失败 icon 放在 TraceId 前，二者合并为一列；整列左对齐。
指标列顺序 输入、输出之后，固定展示端到端耗时、总 Token、总成本。
状态表达 列表状态不再额外占一列，通过 icon 直接在 TraceId 前表达成功/失败。
原型预览：Trace 列表 打开原型
Chapter 04

详情摘要重排

让摘要服务标题与正文，而不是成为第一视觉中心

交互规则
  • 摘要信息带放在 Trace追踪 标题下方。
  • 摘要从独立重卡片，改为弱化视觉权重的信息带。
  • “最终状态”与列表保持同一套成功/失败 icon 表达方式。
原型预览：详情摘要位置与样式 打开详情原型
Chapter 05

过程日志节点头部指标增强

在节点头部一行完成状态与关键指标扫读

目标体验

用户无需先展开正文或跨区域寻找指标，即可在节点头部直接读取状态、耗时、输入、输出、总 Token、成本。

项 交互规则
头部结构 节点标题、状态、耗时、输入、输出、总 Token、成本处于同一行阅读流。
指标顺序 状态之后固定展示：耗时、输入、输出、总 Token、成本。
复制操作 复制当前节点按钮仍保留在头部右侧，不影响指标扫读顺序。
原型预览：节点头部指标展示 打开详情原型
Chapter 06

长日志展开/收起交互优化

只优化交互稳定性，不重复定义折叠阈值本身

问题描述

长日志展开后，用户继续滚动正文时，如果收起入口跟着正文一起离开可视区，会打断阅读与排障节奏。该问题在一条 Trace 中出现多个长日志节点时会被放大。

交互规则
  • 展开和收起使用同一个固定位置，不随状态切换而跳位。
  • 按钮固定在日志块底部区域，正文单独滚动。
  • 多个日志块彼此独立控制，不共享展开/收起状态。
原型预览：长日志收放 打开长日志原型
Chapter 07

验收标准

以评审和走查时可直接验证的行为为准

  • Trace 列表中，时间列前置；TraceId 前出现成功/失败 icon，且与 TraceId 位于同一列、同一对齐方式。
  • 输入、输出之后能稳定看到端到端耗时、总 Token、总成本列。
  • 详情弹窗中，摘要位于 Trace追踪 标题下方，视觉权重明显弱于原先四卡片方案。
  • 详情“最终状态”与列表使用相同的成功/失败 icon 风格。
  • 过程日志节点头部可在一行读取状态、耗时、输入、输出、总 Token、成本。
  • 长日志展开与收起使用同一个底部固定位置；用户滚动正文后仍可直接收起。
  • 一条 Trace 中多个长日志块可分别展开与收起，互不干扰。
